# Pré-processamento e Divisão dos Dados

Antes de alimentar os classificadores de Machine Learning, os dados brutos passam por um fluxo rigoroso de limpeza, transformação e divisão estratégica, garantindo a reprodutibilidade e a integridade estatística dos modelos.

---

## 🛠️ O Fluxo de Pré-processamento
O pipeline de pré-processamento é construído utilizando classes utilitárias do Scikit-Learn (`Pipeline` e `ColumnTransformer`). A estrutura foi desenhada de forma modular para tratar tipos de dados diferentes individualmente.

### 1. Separação de Variáveis e Limpeza
* **Variável Alvo ($y$)**: Isolada a partir do atributo binário `saiu` (convertida explicitamente para tipo inteiro `int`).
* **Atributos de Entrada ($X$)**: O dataframe original é limpo de sua coluna de destino (`saiu`).
* **Tratamento de Strings**: Aplicação de remoção de espaços em branco adicionais no início ou fim (`str.strip()`) para todas as variáveis de tipo texto, limpando eventuais erros de digitação.

### 2. Transformação de Atributos Numéricos
Os atributos numéricos contínuos e discretos (`nivel_satisfacao`, `ultima_avaliacao`, `numero_projetos`, `media_horas_mensais` e `tempo_empresa`) são encaminhados para um pipeline numérico:
* **Padronização (`StandardScaler`)**: Subtrai a média e divide pelo desvio padrão de cada atributo de treino:
  $$x_{scaled} = \frac{x - \mu}{\sigma}$$
* **Por que padronizar?** Modelos lineares como a Regressão Logística e métodos baseados em gradiente sofrem distorções se os atributos estiverem em escalas muito distintas (ex: `nivel_satisfacao` varia entre $0$ e $1$, enquanto `media_horas_mensais` varia entre $90$ e $310$).

### 3. Transformação de Atributos Categóricos
Os atributos categóricos ordinais e nominais (`departamento` e `salario`) são processados por um pipeline específico:
* **Codificação One-Hot (`OneHotEncoder`)**: Cria colunas binárias indicativas para cada categoria textual presente.
* **Prevenção de Multicolinearidade (`drop='first'`)**: Para cada variável categórica, a primeira coluna gerada é descartada (ex: para `salario`, geram-se apenas as colunas binárias representando `medio` e `alto`; a ausência de ambas indica, logicamente, salário `baixo`). Isso impede dependência linear perfeita entre as features transformadas, o que prejudicaria o ajuste dos coeficientes da Regressão Logística.
* **Tratamento de Novas Categorias (`handle_unknown='ignore'`)**: Garante robustez em ambiente de produção. Caso o modelo em produção receba um departamento inédito, o pipeline irá ignorá-lo silenciosamente gerando valores $0$ para todas as colunas de categorias, sem interromper a execução do script.

## 📈 Divisão Estratificada dos Dados (60/20/20)
Para simular com exatidão o comportamento do modelo em dados desconhecidos, a base total de 14.999 registros é dividida em três conjuntos independentes usando amostragem estratificada:

| Partição | Percentual | Quantidade de Registros | Taxa de Turnover | Finalidade |
| :--- | :---: | :---: | :---: | :--- |
| **Treino** | 60% | 8.999 | ~23,6% | Ajuste dos pesos e parâmetros internos dos modelos. |
| **Validação** | 20% | 3.000 | ~23,6% | Seleção do melhor modelo final e ajuste dos hiperparâmetros (GridSearch). |
| **Teste Final** | 20% | 3.000 | ~23,6% | Avaliação final imparcial e cálculo das métricas de desempenho de produção. |

### Por que usar Estratificação (`stratify=y`)?
Como as classes do dataset são desbalanceadas (24% saíram, 76% ficaram), uma divisão puramente aleatória corre o risco de concentrar mais registros da classe minoritária em uma partição do que em outra. A estratificação garante que todas as três partições mantenham exatamente a mesma proporção de ~23,6% de turnover, estabilizando as métricas obtidas em cada etapa.

## 🚫 Prevenção de Data Leakage (Vazamento de Dados)
Um dos erros de metodologia mais severos em Machine Learning é o vazamento de dados, que ocorre quando informações do conjunto de teste ou validação são acidentalmente reveladas ao modelo durante a fase de treino.

No projeto, isso é evitado estruturando o fluxo da seguinte forma:
1. O pipeline de pré-processamento é inicializado.
2. O método **`fit`** do precomputador (que calcula as médias $\mu$ e desvios padrão $\sigma$ para o `StandardScaler`, bem como as categorias conhecidas para o `OneHotEncoder`) é executado **estritamente e exclusivamente** sobre o conjunto de **Treino** (`X_train`).
3. Para os dados de validação (`X_val`) e teste (`X_test`), executa-se apenas o método **`transform`**, aplicando os parâmetros derivados do treino.
4. Isso impede que flutuações, médias ou categorias exclusivas dos dados de teste influenciem o treinamento do modelo, blindando a avaliação final de qualquer viés de otimismo artificial.
